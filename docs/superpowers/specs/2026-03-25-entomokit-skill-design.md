# EntomoKit Skill 化设计文档（面向无命令行经验用户）

**日期**: 2026-03-25  
**状态**: 已评审讨论，待进入实施计划

---

## 1. 背景与目标

当前 `entomokit` 已有完整 CLI 能力，但对无命令行经验的用户，主要痛点不是“命令不存在”，而是：

1. 不知道该选哪个命令与参数组合
2. 输入数据格式不满足隐含约束，导致执行失败
3. 失败后无法根据底层日志快速恢复

本设计目标是将 `entomokit` 封装为可调用 Skill，让用户通过自然语言和少量向导交互完成高频任务，并把“数据准备”前置为首要环节。

---

## 2. 产品定位与范围（MVP）

### 2.1 目标用户

- 第一优先：研究/业务用户（以“拿到结果”为目标，不关心 CLI 细节）
- 第二优先：数据分析初学者（需要有限配置能力）

### 2.2 交互模式

- 采用**混合式交互**：
  - 默认自然语言（低门槛）
  - 关键分叉点使用 2-4 个选项的轻向导（降低歧义）

### 2.3 能力分层

- 深能力（先做 2 条）
  1. 原始数据 -> 标准化处理 -> 可交付产物
  2. 一键分析 -> 报告/图表输出
- 浅能力（配套入口）
  - 数据检查
  - 任务状态查询
  - 结果导出
  - 失败重试

---

## 3. 核心架构与调用链

推荐采用 **Skill + 编排层 + CLI 执行层** 的三层架构。

### 3.1 分层职责

1. **入口层（Skill 对话层）**
   - 意图识别（用户要做什么）
   - 槽位提取（输入路径、目标产物、模型/方法等）
   - 缺失信息最小化追问

2. **编排层（Workflow Orchestrator）**
   - 将请求转换为标准任务计划（task graph）
   - 参数规范化、依赖检查、步骤排序
   - 重试策略、状态机推进、错误分类

3. **执行层（entomokit CLI/API）**
   - 纯确定性执行，不承担对话逻辑
   - 每个步骤输出结构化结果（状态、日志摘录、产物路径）

4. **反馈层（用户可理解输出）**
   - 技术日志翻译为“下一步可执行动作”
   - 展示执行摘要：输入、流程、产物、警告

### 3.2 调用链

`用户请求 -> Skill 解析 -> 编排器计划 -> entomokit 执行 -> 结果聚合 -> 友好反馈`

---

## 4. 数据准备中心（MVP 第一优先级）

数据准备被定义为“成功率杠杆点”。其优先级高于新增命令能力。

### 4.1 设计原则

- 先校验后执行：未经预检的数据不能进入正式流程
- 先示例后上传：用户先看模板和样例，再替换为自己的数据
- 报错必须可修复：错误定位到具体字段/行，提供明确修复动作

### 4.2 关键能力

1. **示例驱动引导**（来源：`data/`）
   - 展示可运行示例目录、最小样本、推荐格式
   - 支持“先跑示例流程再替换真实数据”

2. **输入契约（Data Contract）**
   - 针对每条核心流程定义 schema（字段名、类型、必填、允许值、单位）
   - schema 与命令参数映射保持一一对应

3. **预检三步**
   - `validate`: 必填字段、类型、路径可达性
   - `normalize`: 路径标准化、文件扩展统一、标签标准化
   - `lint`: 异常值、重复值、类分布、缺失统计

4. **字段映射向导**
   - 自动匹配同义列名（例如 `id` -> `sample_id`）
   - 允许人工确认和覆盖

5. **修复辅助输出**
   - 输出“错误摘要 + 行列定位 + 修复模板/示例”
   - 避免直接抛出原始堆栈给终端用户

## 5. 首版范围（5 周节奏）

### Week 1
- 数据准备中心基础：示例展示、模板说明、预检框架

### Week 2
- 编排器骨架：意图 -> 计划 -> 执行 -> 反馈主链路
- 接入第一条深流程

### Week 3
- 接入第二条深流程
- 错误恢复动作库（重试/重映射/修复模板）

### Week 4
- 体验打磨：新手提示语、运行摘要、示例一键跑通

### Week 5
- 端到端回归与脏数据测试
- 文档与演示脚本，冻结 MVP

## 6. 难点与应对

1. **意图歧义**
   - 应对：任务白名单 + 槽位补齐 + 安全默认值

2. **数据质量不稳定**
   - 应对：强制预检 + 映射向导 + 行列级错误定位

3. **失败恢复体验差**
   - 应对：错误码映射到固定恢复动作（重试、修复、降级）

4. **可解释性不足**
   - 应对：输出运行摘要（输入、步骤、参数、产物、警告）

5. **后续扩展复杂化**
   - 应对：插件化任务单元（schema + validator + executor + adapter）

---

## 7. 成功指标（MVP 验收）

1. 新用户 10 分钟内跑通示例流程成功率 >= 90%
2. 真实数据在预检修复后一次成功率 >= 80%
3. 常见失败场景用户自恢复率 >= 85%
4. 核心任务的人机交互轮数 <= 8（目标）
5. 100% 运行产出可追溯摘要（run summary）

---

## 8. 与现有仓库对齐点

1. 现有统一入口 `entomokit <command>` 作为执行层稳定接口
2. `data/` 目录作为“格式示范与最小样本”来源
3. 现有 `README.md` 命令参数说明可转为 Skill 的字段解释知识库
4. 保持“编排层与执行层解耦”，避免把对话逻辑侵入命令实现

---

## 9. 非目标（MVP 不做）

1. 一次性覆盖全部命令与全部高级参数
2. 全图形化前端（重投入，放到后续）
3. 对所有异常自动修复（首版以“可定位 + 可引导修复”为主）
